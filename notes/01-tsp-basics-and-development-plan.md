# TSP: базовое объяснение и план разработки

## 1. Что такое TSP

**TSP = Travelling Salesman Problem = задача коммивояжёра.**

Есть набор городов и расстояния между ними. Нужно построить маршрут, который:

1. стартует из некоторого города;
2. посещает каждый город ровно один раз;
3. возвращается в старт;
4. имеет минимальную суммарную длину.

Пример маршрута:

```txt
A → B → C → D → A
```

Такой замкнутый маршрут называется **tour**, **цикл** или **обход**.

## 2. Почему задача сложная

Если городов мало, можно перебрать все возможные порядки обхода. Но число вариантов растёт факториально.

Для $n$ городов число маршрутов примерно растёт как:

$$
(n-1)!
$$

Примеры:

```txt
5! = 120
10! = 3 628 800
20! ≈ 2.4e18
```

Для challenge-файла с примерно тысячей городов полный перебор невозможен.

## 3. Что значит “комбинаторная оптимизация”

**Комбинаторная** — решение выбирается из огромного числа дискретных вариантов.

В TSP решение — это порядок городов:

```txt
[0, 17, 5, 92, 4, ..., 0]
```

**Оптимизация** — у каждого решения есть стоимость, и нужно найти лучшую.

В TSP:

- решение = порядок обхода городов;
- стоимость = длина маршрута;
- цель = минимизировать длину.

## 4. Матрица расстояний

В проекте входные данные — **distance matrix**, матрица расстояний.

Пример:

```txt
        A   B   C   D
A       0   5   9   4
B       5   0   3   7
C       9   3   0   2
D       4   7   2   0
```

Если маршрут:

```txt
A → B → C → D → A
```

то его длина:

$$
5 + 3 + 2 + 4 = 14
$$

В текущем проекте основной challenge-файл: `data/raw/matrices/M.txt`. Он объявляет и парсится как матрица `1114×1114`, поэтому canonical challenge size — `1114` городов.

## 5. Upper bound

Для задачи минимизации **upper bound** — это длина какого-то реального допустимого маршрута.

Если найден tour длиной `1 000 000`, значит оптимум не может быть больше этого значения:

$$
OPT \le 1\,000\,000
$$

Простая формулировка:

> Upper bound — “я умею проехать вот так; значит оптимум точно не длиннее”.

Чтобы получить upper bound, нужен валидный tour: каждый город посещён ровно один раз, затем возврат в старт.

## 6. Lower bound

**Lower bound** — это число, ниже которого оптимальный маршрут быть не может.

Если доказано, что любой tour имеет длину хотя бы `700 000`, то:

$$
700\,000 \le OPT
$$

Простая формулировка:

> Lower bound — “лучше этого теоретически нельзя, или по крайней мере мы доказали, что нельзя”.

Lower bound требует доказуемо корректного метода. Просто “алгоритм выдал число” недостаточно.

## 7. Gap

Если есть:

```txt
LB = 700 000
UB = 1 000 000
```

то оптимум находится между ними:

$$
700\,000 \le OPT \le 1\,000\,000
$$

**Gap** показывает, насколько близки границы:

$$
gap = \frac{UB - LB}{LB}
$$

Чем меньше gap, тем сильнее результат.

Пример:

```txt
LB = 990 000
UB = 1 000 000
```

Это почти доказанный результат: оптимум отличается от найденного upper bound не более чем примерно на 1%.

## 8. Эвристики для upper bound

**Эвристика** — алгоритм, который не гарантирует оптимум, но часто быстро даёт хороший результат.

### Nearest Neighbor

Идея: каждый раз идти в ближайший ещё не посещённый город.

Плюсы:

- очень просто;
- быстро;
- хороший baseline.

Минусы:

- часто застревает в плохих решениях;
- сильно зависит от стартового города.

### 2-opt

Берём готовый tour и пытаемся улучшить его локально.

Если есть два ребра:

```txt
A → B
C → D
```

можно попробовать заменить их на:

```txt
A → C
B → D
```

Если длина уменьшилась, принимаем замену.

2-opt часто убирает плохие пересечения и резко улучшает простой маршрут.

### 3-opt

Похож на 2-opt, но меняет три ребра за шаг. Обычно мощнее, но медленнее.

### Lin-Kernighan / LKH-подобные методы

Сильные практические эвристики для TSP. Часто дают почти оптимальные решения на больших задачах.

## 9. Методы для lower bound

### MST bound

**MST = Minimum Spanning Tree = минимальное остовное дерево.**

Остовное дерево соединяет все города без циклов. Это более слабое требование, чем TSP-tour, поэтому его стоимость может дать нижнюю границу.

Интуиция:

> Чтобы пройти все города циклом, нужно как минимум как-то их соединить. Дерево — более слабое требование, значит оно не дороже оптимального tour.

### 1-tree bound

Более сильная нижняя граница.

Идея:

1. выбрать один специальный город;
2. построить MST по всем остальным городам;
3. добавить два самых дешёвых ребра из специального города.

Получается структура, похожая на TSP-cycle, но ограничение всё ещё слабее настоящего TSP. Поэтому это может быть lower bound.

### Assignment relaxation

TSP можно представить как выбор входящих и исходящих рёбер:

- из каждого города выйти ровно один раз;
- в каждый город войти ровно один раз.

Если убрать требование “один общий цикл”, можно получить несколько маленьких циклов. Это уже не настоящий TSP, но более лёгкая задача. Её оптимум даёт lower bound.

## 10. Relaxation

**Relaxation = ослабление задачи.**

Берём исходную задачу и убираем часть ограничений.

TSP требует:

1. посетить все города;
2. каждый ровно один раз;
3. вернуться в начало;
4. получить один общий цикл.

Если убрать часть ограничений, задача становится легче. Для задачи минимизации оптимум ослабленной задачи не больше настоящего оптимума:

```txt
relaxed optimum ≤ TSP optimum
```

Поэтому relaxation часто даёт lower bound.

## 11. Минимальный словарь

| Термин | Смысл |
|---|---|
| City / node / vertex | город, вершина |
| Edge | ребро между двумя городами |
| Distance / cost / weight | расстояние или стоимость ребра |
| Tour | замкнутый маршрут через все города |
| Feasible solution | допустимое решение, не нарушающее правил |
| Objective | целевая функция: что минимизируем |
| Upper bound / UB | длина найденного допустимого tour |
| Lower bound / LB | доказанная нижняя граница на оптимум |
| Gap | разрыв между UB и LB |
| Heuristic | быстрый практический алгоритм без гарантии оптимума |
| Relaxation | ослабленная задача, дающая bound |
| MST | минимальное остовное дерево |
| 1-tree | MST по всем кроме одной вершины + два ребра из неё |

## 12. Что нужно уметь объяснить лектору

К 15-минутному разговору нужно уметь объяснить:

1. как читалась матрица;
2. какая размерность использовалась;
3. как проверялась корректность tour;
4. какой лучший upper bound найден;
5. каким алгоритмом он найден;
6. какой lower bound найден;
7. почему lower bound корректен;
8. какой gap между bounds;
9. почему результат сравним с другими: использована та же матрица.

## 13. Базовый план разработки

### Шаг 1 — понять данные

Проверить:

- формат `M.txt`;
- размерность;
- симметричность матрицы;
- нули на диагонали;
- отсутствие странных отрицательных значений;
- canonical challenge size: `1114` городов; матрица должна иметь `1114` строк и `1114` столбцов.

### Шаг 2 — базовая инфраструктура

Сделать в `src/`:

- загрузчик матрицы;
- функцию вычисления длины tour;
- валидатор tour;
- сохранение результата с метаданными;
- независимую перепроверку длины сохранённого маршрута.

### Шаг 3 — первый upper bound

Начать с простого и воспроизводимого pipeline:

1. nearest neighbor from multiple starts;
2. улучшение через 2-opt;
3. сохранение лучшего tour в `results/best/`.

Минимальные метаданные результата:

- input file;
- length;
- tour;
- algorithm;
- parameters;
- seed;
- date;
- command used.

### Шаг 4 — первый lower bound

Начать с:

1. MST bound;
2. 1-tree bound.

Для каждого lower bound сохранить:

- значение;
- метод;
- входную матрицу;
- короткое объяснение, почему это корректная нижняя граница.

### Шаг 5 — улучшать обе стороны

Upper bound:

- больше стартов nearest neighbor;
- randomized starts;
- 2-opt быстрее и аккуратнее;
- 3-opt;
- LKH-подобные идеи.

Lower bound:

- лучший 1-tree;
- subgradient optimization для 1-tree;
- assignment relaxation;
- сравнение разных lower-bound методов.

### Шаг 6 — готовить объяснение

Параллельно вести `results/best/summary.md`:

```txt
Best UB: ...
Method: ...
Best LB: ...
Method: ...
Gap: ...
Input: ...
```

К финалу должно быть не только число, но и короткая история:

- от какого baseline начали;
- что улучшило upper bound;
- что улучшило lower bound;
- почему итоговые числа корректны.

## 14. Главная формула понимания

> TSP — это поиск кратчайшего цикла по всем городам. Полный перебор невозможен, потому что вариантов слишком много. Поэтому мы ищем хороший реальный маршрут — это upper bound. И отдельно доказываем, что лучше некоторого числа нельзя — это lower bound. Чем ближе upper и lower bound, тем сильнее результат.
